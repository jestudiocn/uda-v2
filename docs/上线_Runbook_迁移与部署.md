# UDA-V2 上线 Runbook（本机 → ECS，含 V1 数据迁移）

本文档固化你已同意的上线顺序，并与 ECS 生产约束对齐：**禁止覆盖 `.env`、`storage`；代码以 Git 为准；数据库变更可控。**

---

## 环境与约束（必读）

| 项 | 要求 |
|----|------|
| 站点根目录 | `/var/www/uda-v2`，Nginx `root` → `public`（勿改路径） |
| 代码更新 | `git pull origin main`（GitHub Actions 已配置），**禁止** `git clone` 覆盖整目录 |
| **禁止覆盖** | `/var/www/uda-v2/.env`、`/var/www/uda-v2/storage`（整棵） |
| 权限 | `chown -R www-data:www-data /var/www/uda-v2`；`storage` 等需可写 |
| Actions 现状 | 仅 `git pull` + `chown`，**无** `migrate`、**无** `composer`；依赖变更后可在 ECS **手工**执行 `composer install --no-dev --optimize-autoloader` |
| 正式上线后 | **禁止**整库覆盖生产库；变更仅 **增量 SQL / 受控脚本** |

---

## 阶段 0：每次动库/动机前

- [ ] 本机：`mysqldump` 全库或关键库备份，文件名带日期。
- [ ] ECS：备份 `.env`（`cp .env .env.bak.YYYYMMDD`）、`tar` 打包 `storage`（尤其上传、POD 等）。
- [ ] 记录「最高管理员」账号：`users.id` / `username`（本机清测试数据时保留依据）。
- **本次上线已备案（2026-05-02）**：最高管理员 `users.id = 1`，`username = admin`（阶段 1 清测试数据时仅保留该账号为最高管理员依据）。

---

## 阶段 1：本机 V2 清测试数据（保留最高管理员）

- [ ] 在**已备份**前提下，删除/清空测试业务数据，**保留唯一最高管理员**及其登录所依赖的行（角色、权限中间表等按实际外键处理）。
- [ ] 本机 smoke：管理员可登录、核心菜单可进。

---

## 阶段 2：ECS 去掉旧测试 V2（不动主机级配置）

- [ ] **数据库**：删除/重建测试库或清空业务表（按你接受程度），**不**依赖 Actions 自动执行。
- [ ] **代码目录**：**不要** `rm -rf /var/www/uda-v2` 再 clone。保留 `.env` 与 `storage`，仅通过后续 `git pull` 更新代码树。
- [ ] 若必须清理工作区：在目录内用 git 恢复到干净状态（如 `fetch` + `reset` 到目标分支），仍保留未入库的 `.env` 与 `storage`。

---

## 阶段 3：本机最新代码与「基线库」上 ECS

**拆成两件事执行：**

### 3A 代码

- [ ] 本机：`commit` → `push` `main` → 触发 Actions → ECS `git pull`。
- [ ] 若 `composer.lock` 有变：SSH 至 ECS 在项目根 **手工** `composer install --no-dev --optimize-autoloader`（勿把 composer 写进自动 migrate 的流程）。

### 3B 数据库（本次允许整库替换测试库）

- [ ] 本机导出 V2 库 SQL（结构 + 数据，或你当前策略下的 dump）。
- [ ] 导入 ECS MySQL 目标库。
- [ ] **仅**在 ECS 上按需编辑 `.env` 中的 **DB 连接**（主机/库名/用户/密码），**禁止**用本机 `.env` 整文件覆盖服务器 `.env`（避免 `APP_KEY`、域名等与线上一致性被破坏）。

---

## 阶段 4：从轻量机拉取「当前为止」的源数据

- [ ] 下载 V1（或现行）库导出 / 指定表导出到本机。
- [ ] 与已有 `finance.sql` 等文件区分版本，命名带日期与环境。

---

## 阶段 5：本机完成 V1 → V2 迁移与测试（核心）

- [ ] 依赖顺序建议：**用户与 ID 映射** → 行事历 → 财务 → 问题订单（及中间主数据）。
- [ ] 维护 **V1 表/列 → V2 表/列** 对照与 **旧 ID → 新 ID** 映射（至少用户）。
- [ ] 本机验收清单（示例）：登录与权限、行事历列表/编辑、财务关键录入与列表、问题订单、派送关键页、导出/上传。
- [ ] 通过后本机再 **dump 一份「已迁移的 V2 库」** 作为上 ECS 的输入。

---

## 阶段 6：已迁移数据灌入 ECS 做联调测试

- [ ] 将阶段 5 产出的 SQL 导入 ECS（覆盖当前 ECS 测试库或按你命名的新库），`.env` 指向该库。
- [ ] 核对 PHP/MySQL 版本、`sql_mode` 与字符集，减少排序与日期边界问题。
- [ ] HTTPS、`storage/logs`、上传路径写权限、关键业务流程再走一遍。

---

## 阶段 7：正式上线与后续维护

- [ ] DNS/业务侧确认流量与维护窗口（若从 V1 主机切换）。
- [ ] 宣布上线后：**代码** = `main` + GitHub Actions；**生产库** = 仅增量变更 + 变更记录/回滚预案。
- [ ] 本机持续开发：功能在分支开发，**合并至 `main` 再 push`**；勿期待 Git 自动同步数据库。

---

## 回滚思路（简记）

- **代码**：`git` 回到上一已知良好 commit 再 `pull`。
- **数据**：依赖阶段 0 的 dump；生产阶段**严禁**无备份整库覆盖。
